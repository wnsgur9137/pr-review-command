# /commit

스테이징된(또는 사용자가 승인한) 변경을 분석하고, **이 레포의 기존 커밋 컨벤션**에 맞춰
gitmoji + Conventional Commits 형식의 커밋 메시지를 생성·실행한다.

기본 동작:
- **gitmoji**: 기본 사용하되, **감지된 컨벤션이 우선**한다(아래 3단계 참고).
- **Co-Authored-By**: 기본 **유지**. `--no-coauthor`로만 제거.
- **push는 절대 자동 실행하지 않는다.** 커밋까지만 한다.

## 입력 인자

사용자가 넘긴 인자: `$ARGUMENTS`

각 옵션은 짧은 별칭을 지원한다. **소문자 = 동작 켜기, 대문자 = 끄기/제거(`--no-*`)** 규칙.

- `-G`, `--no-gitmoji`: gitmoji를 절대 붙이지 않음 (컨벤션·기본값보다 우선).
- `-g`, `--gitmoji`: gitmoji를 강제로 붙임 (컨벤션이 이모지를 안 써도 강제, `--no-gitmoji`보다는 약함).
- `-C`, `--no-coauthor`: 커밋 메시지에서 `Co-Authored-By: Claude ...` 푸터 제거.
- `-a`, `--all`: 스테이징된 변경이 없을 때 확인 없이 `git add -A` 후 진행.
- `-e`, `--amend`: 새 커밋 대신 **직전 커밋을 수정**(amend).
- `-S`, `--no-split`: 변경이 여러 성격이어도 커밋 분리를 제안하지 않고 단일 커밋으로 진행.
- `-y`, `--yes`: 커밋 전 미리보기 확인을 생략하고 바로 커밋.
- `-h`, `--help`: 아래 **Help Mode**의 도움말을 출력하고 종료(아무것도 커밋하지 않음).

짧은 별칭은 묶어 쓸 수 있다(예: `-ay` = `--all --yes`, `-Gy` = `--no-gitmoji --yes`).
인자가 충돌하면(`-G`/`--no-gitmoji`와 `-g`/`--gitmoji` 동시) `--no-gitmoji`를 우선한다.

## 실행 절차

다음 순서대로 정확히 수행한다. 각 단계 결과를 한 줄로 보고하며 진행한다. 한국어로 응답한다.

### 0. Help Mode (`-h` 또는 `--help`)

인자에 `-h` 또는 `--help`가 포함되어 있으면 **다른 단계를 수행하지 않고** 아래 도움말만 출력하고 즉시 종료한다(스테이징·커밋 등 어떤 변경도 하지 않는다).

```
=== /commit 도움말 ===

Usage: /commit [options]
Example: /commit
         /commit -ay
         /commit --gitmoji --no-coauthor

변경사항과 이 레포의 기존 커밋 컨벤션을 분석해 gitmoji·Conventional
Commits 형식의 커밋을 생성합니다. push는 절대 자동 실행하지 않습니다.

-- Options --  (소문자=켜기, 대문자=끄기/제거, 묶어쓰기 가능: -ay)
  -G, --no-gitmoji    gitmoji를 절대 붙이지 않음 (최우선)
  -g, --gitmoji       컨벤션이 이모지를 안 써도 gitmoji 강제
  -C, --no-coauthor   Co-Authored-By: Claude ... 푸터 제거 (기본은 유지)
  -a, --all           스테이징 변경이 없을 때 git add -A 후 진행
  -e, --amend         새 커밋 대신 직전 커밋 수정 (푸시됐으면 경고)
  -S, --no-split      변경이 섞여 있어도 분리 제안 없이 단일 커밋
  -y, --yes           커밋 전 미리보기 확인 생략 (민감 파일 경고는 유지)
  -h, --help          이 도움말을 표시합니다

-- gitmoji 우선순위 --
  1. -G/--no-gitmoji          → 미사용 (최우선)
  2. -g/--gitmoji             → 사용
  3. 인자 없음                → 감지된 레포 컨벤션을 따름
                                (이모지 일관 사용 시 사용, 아니면 미사용)
  4. 컨벤션 불명확/히스토리 거의 없음 → 기본값 ON

-- 동작 절차 --
  1. 변경/스테이징 상태 파악 (스테이징 없으면 확인, 보호 브랜치 경고)
  2. 기존 커밋 컨벤션 분석 (형식·이모지·언어·본문·scope·co-author)
  3. gitmoji 사용 여부 결정 (필요 시 brew/npm으로 CLI 설치)
  4. 커밋 분리 판단 (여러 논리 단위면 분리 제안)
  5. 메시지 생성 → 6. 미리보기 확인 → 7. 커밋 → 8. 보고
```

### 1. 변경 / 스테이징 상태 파악

```bash
git rev-parse --abbrev-ref HEAD
git status --short
git diff --cached --stat        # 스테이징된 변경
git diff --stat                 # 스테이징 안 된 변경
```

- **스테이징된 변경이 있으면** 그것만 대상으로 진행.
- **스테이징된 변경이 없으면**:
  - `--all`이 있으면 `git add -A` 실행 후 진행.
  - 없으면 **AskUserQuestion**으로 "스테이징된 변경이 없어. 전체(`git add -A`)를 스테이징하고 커밋할까?"를 묻는다. 거부하면 중단.
- 작업 트리에 변경이 전혀 없으면(스테이징·미스테이징 모두 0) "커밋할 변경이 없다"고 알리고 중단.
- 현재 브랜치가 `main`/`master`/`develop` 같은 **보호 브랜치면 경고**하고 진행 여부를 확인한다(AskUserQuestion). 중단해도 그대로 둔다.

### 2. 컨벤션 분석 (가장 중요)

최근 커밋을 읽어 이 레포의 컨벤션을 추론한다.

```bash
git log -30 --format='%s'        # 제목 패턴
git log -10 --format='%B'        # 본문/푸터 패턴
```

다음을 판별한다.

- **형식**: Conventional Commits(`type:` 또는 `type(scope):`)인가, 단순 문장인가.
- **이모지 사용 여부**: 제목 앞에 gitmoji/이모지가 **일관되게** 붙는가. (절반 이상이면 "사용", 거의 없으면 "미사용")
- **언어**: 제목이 주로 한국어인가 영어인가.
- **본문 사용**: 커밋에 본문(상세 설명)이 자주 붙는가, 제목만 쓰는가.
- **scope 패턴**: `feat(scope):`처럼 scope를 쓰는가.
- **Co-Authored-By 패턴**: 기존 커밋에 co-author 푸터가 있는가.

> 이 레포는 현재 `chore:`, `feat:`, `docs:` 같은 **Conventional Commits(이모지 없음)** 컨벤션이다. 이 경우 기본적으로 gitmoji를 **생략**한다.

### 3. gitmoji 사용 여부 결정 (우선순위)

다음 우선순위로 최종 결정한다.

1. `--no-gitmoji` 인자가 있으면 → **사용 안 함** (최우선).
2. `--gitmoji` 인자가 있으면 → **사용**.
3. 위 인자가 없으면 → **2단계에서 감지된 컨벤션을 따른다.**
   - 기존 커밋이 이모지를 일관되게 쓰면 → 사용.
   - 기존 커밋이 이모지를 안 쓰면 → **사용 안 함.**
4. 컨벤션이 불명확하거나(신규 레포/혼재) 커밋 히스토리가 거의 없으면 → **기본값 ON(사용).**

#### gitmoji CLI 확인 (사용하기로 결정된 경우에만)

```bash
command -v gitmoji
```

- 설치돼 있으면 그대로 사용(아래 매핑 표는 직접 적용).
- **설치돼 있지 않으면** AskUserQuestion으로 묻는다:
  > "gitmoji를 쓰려면 CLI가 필요해. 설치할까? (① `npm i -g gitmoji-cli` 설치  ② 이번엔 gitmoji 없이 진행)"
  - "설치" 선택 시: **`brew install gitmoji`** 실행(Homebrew 우선). `brew`가 없으면 `npm i -g gitmoji-cli`로 폴백. 둘 다 실패하면 사유 보고 후 gitmoji 없이 진행.
  - "없이 진행" 선택 시: 이 커밋은 gitmoji 없이 생성.

> 참고: 실제 메시지에는 **유니코드 이모지 문자**(예: `✨`)를 직접 넣는다. `:sparkles:` 코드 형식이 아니다. CLI는 매핑 참고/설치 확인 용도로만 쓰고, 메시지 조립은 직접 한다.

#### 타입 → gitmoji 매핑 (유니코드)

| type | emoji | type | emoji |
|---|---|---|---|
| feat | ✨ | test | ✅ |
| fix | 🐛 | build | 📦️ |
| docs | 📝 | ci | 👷 |
| style | 🎨 | revert | ⏪️ |
| refactor | ♻️ | init | 🎉 |
| perf | ⚡️ | remove | 🔥 |
| chore | 🔧 | security | 🔒️ |
| deps(추가) | ➕ | deps(업) | ⬆️ |
| breaking | 💥 | wip | 🚧 |

### 4. 커밋 분리 판단

스테이징된 변경이 **여러 논리 단위**(예: 기능 추가 + 무관한 버그 수정 + 설정 변경)로 섞여 있는지 판단한다. 판단 근거:

- 변경 파일이 서로 다른 모듈/관심사에 걸쳐 있고, 성격(type)이 명확히 다른가.
- 한 커밋으로 묶으면 메시지의 type 하나로 설명이 안 되는가.

분리가 적절하다고 판단되면:

- **AskUserQuestion**으로 분리 제안을 보여준다. 예:
  ```
  변경이 2개 논리 단위로 보여. 어떻게 할까?
  ① 분리: [커밋1] ✨ feat: A 기능 (fileA, fileB) / [커밋2] 🔧 chore: 설정 변경 (.mcp.json)
  ② 단일 커밋으로 진행
  ③ 취소
  ```
- "분리" 선택 시: 각 그룹별로 `git reset`(스테이징 해제) 후 그룹 파일만 `git add`하여 5~7단계(메시지 생성·미리보기·커밋)를 **그룹마다 반복**한다. 그룹 분리는 파일 단위로만 하고, 한 파일 내 hunk 분리(`git add -p`)는 사용자가 명시적으로 요청할 때만 한다.
- "단일" 선택 시: 아래 단일 커밋 흐름으로 진행.

다음 경우 분리 판단을 건너뛰고 단일 커밋으로 간다.

- `--no-split` 인자가 있을 때.
- `--amend`일 때(직전 커밋 수정이므로 분리 무의미).
- 변경이 명백히 하나의 관심사일 때.

### 5. 커밋 메시지 생성

스테이징된 diff와 2단계 컨벤션을 근거로 작성한다.

```bash
git diff --cached              # 메시지 작성 근거. 길면 핵심만.
```

**제목(첫 줄)**
- 형식: `[<emoji> ]<type>[(<scope>)]: <요약>`
  - gitmoji 사용 시 맨 앞에 이모지 + 공백. 미사용 시 이모지 생략.
  - `type`은 변경 성격으로 결정(feat/fix/docs/refactor/chore/test/perf/style 등).
  - `scope`는 변경 파일 경로에서 추론(예: 한 디렉터리에 집중되면 그 이름). 불명확하면 생략.
- 언어는 2단계에서 감지한 레포 언어를 따른다.
- 명령형, **50자 이내 권장**(최대 72자). 마침표로 끝내지 않는다.
- 추상적 표현 금지("개선", "수정함"만 적지 말 것). 무엇을/왜를 구체적으로.

**본문(선택)**
- 변경이 작고 제목으로 충분하면 본문 생략(레포가 제목만 쓰는 컨벤션이면 더욱).
- 변경이 크거나 "왜"가 필요하면, 제목과 빈 줄 뒤에 불릿으로 핵심 변경·이유를 적는다. 한 줄 72자 권장.
- 브랜치명에 이슈 번호 패턴(`feat/ISSUE-123-...`, `123-...` 등)이 있으면 본문 끝 또는 푸터에 `Refs: #123` 형태로 참조(불확실하면 넣지 않는다).

**푸터 — Co-Authored-By**
- 기본적으로 다음 줄을 메시지 맨 끝에 **유지**한다. 현재 작업 중인 모델 버전을 명시한다(예시는 Opus 4.8).
  ```
  Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
  ```
- `--no-coauthor` 인자가 있으면 이 줄을 **넣지 않는다**.

### 6. 미리보기 + 사용자 확인

커밋을 실행하기 **전에** 최종 메시지와 대상 파일을 보여주고 확인받는다.

```
─ 커밋 미리보기 ─────────────────
<최종 커밋 메시지 전문>
────────────────────────────────
대상: N개 파일 (+추가 / -삭제)
gitmoji: 사용 | 미사용 (사유: 컨벤션/옵션)
```

- AskUserQuestion으로 "이 메시지로 커밋할까? (① 커밋  ② 메시지 수정  ③ 취소)"를 묻는다.
- "수정" 선택 시 사용자 피드백을 반영해 다시 미리보기.
- **`--yes`(또는 `-y`) 인자가 있으면 이 단계를 생략하고 바로 7단계로 진행**한다. 단, 민감 파일·시크릿 경고(안전 규칙)는 `--yes`여도 건너뛰지 않는다.

### 7. 커밋 실행

확인되면 실행한다. 메시지는 HEREDOC으로 안전하게 전달한다.

```bash
git commit -F - <<'EOF'
<최종 메시지>
EOF
```

- `--amend` 인자가 있으면 `git commit --amend -F - <<'EOF' ... EOF` 사용.
  - 직전 커밋이 **이미 원격에 푸시된 경우**(`git log origin/<branch>..HEAD`로 확인) amend는 force push가 필요하므로 **경고**하고 진행 여부를 확인한다.
- 메시지 본문에 `'EOF'`가 포함되면 다른 구분자(`'OMC_COMMIT_EOF'` 등)로 바꾼다.
- **pre-commit hook이 실패하면** 그 출력을 보여주고 중단한다. `--no-verify`는 사용자가 명시적으로 요청할 때만 사용한다.

### 8. 사후 보고

```
✅ 커밋 완료
- SHA: <짧은 해시> <제목>
- 변경: M개 파일 (+추가 / -삭제)
- gitmoji: 사용(✨) | 미사용
- Co-Authored-By: 포함 | 제외
- 다음 단계: git push (자동 실행하지 않음)
```

## 안전 규칙

- **push, force push, 브랜치 checkout/생성, reset/rebase 등은 절대 자동 실행하지 않는다.** 이 커맨드는 커밋까지만 한다.
- `.env`, `*.key`, `*.pem`, `credentials*`, `id_rsa` 같은 **민감 파일이 스테이징에 포함**돼 있으면 커밋 전에 경고하고 진행 여부를 확인한다.
- API 키·토큰·비밀번호 같은 시크릿이 diff에 보이면 경고한다. 메시지에 시크릿을 절대 인용하지 않는다.
- 보호 브랜치(`main`/`master`/`develop`)로의 직접 커밋은 1단계에서 경고한다.
- 사용자가 명시하지 않은 파일을 임의로 스테이징하지 않는다(`--all`/사용자 승인이 있을 때만 전체 add).

## 출력 톤

- 진행 단계마다 한 줄씩 짧게 알린다. 장황한 설명 금지.
- 한국어로 응답.
